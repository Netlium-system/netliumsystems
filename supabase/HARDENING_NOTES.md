## Withdrawal Balance Model (Issue 2)

`request_wallet_withdrawal()` records withdrawals as `status = 'pending'`.  
The database enforces: authenticated caller, wallet ownership, amount > 0, non-empty asset/network/destination.  
**The database does NOT atomically check available balance.**

**Reason:** The `wallet_transactions` ledger table has no balance column. Deriving available balance (sum of completed deposits minus sum of completed/pending withdrawals) across concurrent writes without a dedicated balance row would require a serializable transaction or an explicit advisory lock outside the current function scope.

**Implication:** This is a non-binding withdrawal request. Operations staff must verify available balance before approving (changing status to `completed`) or rejecting (status `failed`/`cancelled`). The system prevents forged ledger entries and invalid inputs, but does not prevent over-requests. A future migration should add an atomic balance reservation model if immediate enforcement is required.

## Document Storage Model (Issue 11)

The `documents` private bucket is **institution-issued only**.

- Documents (statements, reports, compliance files) are generated by backend/service-role processes.
- RLS: `documents_select_own` — users can SELECT their own document metadata.
- Storage RLS: `documents_storage_read_own` — users can read objects in their own folder.
- **No authenticated upload policy exists and none should be added.** Object creation requires `service_role`.
- This is verified by `supabase/migrations/20260712020000_documents.sql`, which adds only SELECT policies and no INSERT policy on `public.documents` or `storage.objects`.
